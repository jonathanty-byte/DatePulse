## Contraintes techniques rappelées
- Format B Tinder = dailyOnly (pas de timestamps horaires pour swipes)
- Hourly chart basé sur message timestamps (proxy) — disclaimer maintenu
- "Tes meilleurs jours" utilise les totaux quotidiens groupés par jour de semaine — faisable en dailyOnly
- 100% client-side, aucun coût serveur pour le Wrapped

## Dette technique acceptée
1. enrichMatchesWithMessages = fuzzy match ±24h, collisions possibles si 2 matches le même jour. Tests Vitest atténuent le risque.
2. Benchmarks qualitatifs = wording subjectif ("zone optimale"), pas de source chiffrée FR. Risque de perception manipulatoire si mal formulé.
3. RadarChart axes = approximations (pas de ground truth pour "Régularité" ou "Timing"). Les axes doivent être transparents dans l'UI.
4. active_time Tinder = format inconnu, potentiellement absent. Fallback sur estimation 2s/swipe existante.
5. Share image Canvas API = rendu non-identique cross-browser (Safari/Chrome). Acceptable pour MVP.
